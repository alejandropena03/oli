Fundamentos
El TDD es el plano.
El código es la construcción.
Un Technical Design Document define qué se va a construir y por qué antes de tocar el código. Si algo no está aquí, no se construye.
Regla 01
Nunca borrar ADRs
Las decisiones de arquitectura se marcan como deprecated pero nunca desaparecen. En 6 meses, sabrás exactamente por qué algo está como está.
Regla 02
Los schemas son contratos
Si cambias la forma de un dato, actualizas todos los módulos que lo usan. No hay sorpresas en producción.
Regla 03
Prueba en papel primero
Cada caso de uso se describe de punta a punta antes de codificar. Si no puedes describirlo, la arquitectura está incompleta.

Entiende primero
Oli pregunta antes de asumir. Máximo 2 preguntas, cada una con propósito. Si no entiende bien la intención, todo lo demás sale mal.
3 intentos antes de pedir ayuda
Si algo falla, Oli lo intenta solo 3 veces con estrategias distintas. Solo escala al founder cuando agotó las opciones y llega con contexto completo.
No entrega sin validar
El resultado se verifica contra los criterios de éxito definidos al inicio. Si no los cumple, repara antes de entregar. Sin outputs decorativos.
La máquina de estados
Una misión puede estar
en 18 estados distintos.
Cada estado tiene una lógica clara. Solo el Mission Kernel puede cambiar el estado de una misión. Los trabajadores internos reportan resultados — no deciden estados.

Clarificación de intención — corregido
Oli pregunta todo
lo que necesite.
No hay límite fijo.
Un límite arbitrario de "2 preguntas" lleva a Oli a asumir cosas que no debería. Los humanos son malos optimizando sus prompts — Oli debe compensar eso.
Antes de preguntar, Oli verifica
1. ¿Está en la memoria del usuario? → usa directamente
2. ¿Está en la memoria de empresa? → usa directamente
3. ¿Se puede inferir con alta confianza? → usa con nota
4. ¿Si lo asumo mal, el costo de corrección es bajo? → asume y avisa
5. ¿Si lo asumo mal, el resultado es inutilizable? → pregunta
Lo que sí tiene límite: la calidad
Cada pregunta debe ser necesaria y tener un propósito explícito. Oli nunca pregunta lo mismo dos veces en la misma misión. Con el tiempo, pregunta menos porque conoce mejor al founder.
❌ Incorrecto — pregunta lo que puede inferir
Founder: "Oli, prepara el reporte"
Oli: "¿Qué tipo de reporte?"
(la memoria dice que siempre hace el reporte semanal de ventas)
✅ Correcto — usa memoria, pregunta solo lo nuevo
Founder: "Oli, prepara el reporte para el pitch de mañana"
"Entiendo — el reporte de ventas en formato ejecutivo. Necesito saber: ¿qué período cubre (este mes, Q1, YTD)? ¿A quién va dirigido específicamente?"
(pregunta 2 cosas porque sin ellas el resultado sería incorrecto)
Cuando algo falla — corregido
Oli diagnostica.
No reintenta ciegamente.
El ciclo de troubleshooting correcto no tiene límite de intentos fijo. Continúa mientras Oli tenga nuevas estrategias y cada intento produzca nueva información.
1
Diagnosticar
¿Qué falló exactamente? ¿Cuál es la causa raíz? No el síntoma — la causa.
2
Clasificar el error
¿Transitorio? ¿Autenticación? ¿Recurso no encontrado? ¿Error de lógica? ¿Herramienta caída?
3
Aplicar estrategia según tipo
Transitorio → retry con backoff. Auth → renovar token. Herramienta caída → alternativa. Lógica → replantear el step.
4
Verificar y repetir si hay info nueva
Si falló pero produjo nueva información → nueva iteración con estrategia distinta.
!
Escalar cuando el ciclo no converge
La solución requiere acceso que solo el founder tiene, o el costo de continuar supera el beneficio estimado.
Lo que Oli presenta al escalar
Diagnóstico: qué falló y por qué
Intentos: qué estrategia aplicó y qué resultado dio
Límite: por qué no puede resolverlo solo
Opciones: Opción A / B / C con resultado esperado de cada una
Los errores se aprenden
Los patrones de error y sus soluciones se guardan en Mission Memory. La próxima vez que ocurra el mismo tipo de error, Oli llega directo a la solución correcta. El troubleshooting mejora con el uso.
La feature más importante — ADR-003
Memoria infinita.
RAG desde el día uno.
Sin RAG, Oli empieza cada misión desde cero. Es un chatbot con mejor UI. Con RAG, Oli recuerda todo lo que ha aprendido — para siempre — y recupera exactamente lo relevante para cada misión.
Capa 1
Contexto activo (RAM)
Misión actual + lo recuperado del vector store. ~10-50K tokens. Dura la sesión.
Capa 2
Vector store — ChromaDB (V0) → Qdrant (V3+)
Todo lo que Oli aprendió, en embeddings. Búsqueda semántica. Crece sin límite. Es el núcleo del RAG.
Capa 3
SQLite / PostgreSQL — queries exactas
"Todas las misiones de este mes", "costo por tipo". Complementa el vector store.
Capa 4
Cold storage — historial archivado
Datos de más de N meses. JSON comprimido. Recuperable on-demand si se necesita contexto muy antiguo.
Cómo funciona RAG en cada misión
Founder: "Oli, prepara la propuesta para Acme Corp"
Antes de hacer nada, Oli hace retrieval:
→ "Acme Corp" → reuniones, pain points, sector
→ "propuesta" → formato, propuestas que funcionaron
→ "Acme industria" → research previo, competidores
Resultado: 5-15K tokens de contexto exactamente relevante inyectados al LLM. No todo — solo lo que importa para esta misión.
La memoria es el moat
Un competidor puede copiar la UI, los tools, el Mission Kernel. No puede copiar 2 años de contexto acumulado de un usuario. Cuanto más usa Oli el founder, más valioso se vuelve y más difícil es irse.
Model routing — ADR-016
Oli decide qué modelo usar.
Local cuando puede.
API cuando lo vale.
Tier 1 — Modelo local (costo $0)
Tasks simples: intent, clasificación, routing
Llama 3.1 8B / Mistral 7B via Ollama. Si hay GPU: gratis. Sin GPU: Haiku como fallback barato.
Tier 2 — GPU potente o Haiku
Razonamiento medio: planificación, síntesis
Con GPU 48GB: Llama 3.1 70B o Qwen 2.5 72B. Sin GPU suficiente: Claude Haiku o GPT-4o-mini.
Tier 3 — Claude Sonnet (siempre API)
Tasks complejas: arquitectura, código profundo
Los modelos locales no llegan a este nivel todavía. Claude Sonnet es la opción correcta para decisiones de alta consecuencia.
GPU alquilada — el cálculo
Sin GPU — 100% API (100 misiones/mes):
~$5/mes en modelos
Con GPU alquilada (RunPod A40, ~$0.50/hr):
~5 hrs/mes de uso → $2.50 GPU
+ $1 API solo para Tier 3
= $3.50/mes — más barato Y más privado
Break-even: ~30-40 misiones/mes
Privacy mode — el usuario elige
local_only: todo en GPU propia. Nada va a Anthropic/OpenAI. Trade-off: calidad menor en Tier 3.
hybrid (default): simples → local, complejos → API. Mejor balance.
cloud_ok: todo puede ir a APIs. Más fácil, más caro.

AirPlay (macOS)
Para Apple TV o Smart TVs compatibles. Latencia variable. Más dependiente del hardware.
En todos los casos: el TV nunca tiene credenciales. Solo recibe el estado actualizado de la app desktop via IPC local o Cast. Las credenciales viven siempre en el OS Keychain del computador.
